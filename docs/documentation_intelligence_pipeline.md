# Documentation Intelligence Pipeline

This file is the operating instruction for crawling EduBoost documentation and turning it into higher-level artifacts.

## Purpose

EduBoost has many source documents spread across `docs/`, `audits/`, `reports/`, root Markdown files, release evidence, and generated API references. The documentation intelligence pipeline keeps those sources discoverable, classifies them, identifies gaps, and creates a grounded plan for producing formal deliverables.

The first deliverable is generated by:

```bash
python3 scripts/docs_inventory.py
```

Generated outputs live in:

```text
docs/generated/documentation_intelligence/
```

Do not edit generated inventory outputs by hand. Refresh them from the script.

## Pipeline Stages

1. Local repository discovery

Run `scripts/docs_inventory.py` first. It crawls known documentation roots, classifies files, extracts titles, headings, links, dates, status signals, local references, evidence references, and gap signals.

2. Document conversion

Use conversion only when the inventory shows binary or external documents that need text extraction.

- Use MarkItDown for lightweight conversion to Markdown when speed and broad file support matter.
- Use Docling when richer PDF/DOCX/PPTX structure matters, especially tables, layout, reading order, OCR, formulas, and document sections.
- Keep converted Markdown as an intermediate artifact, not as the source of truth unless explicitly promoted.

Example optional runs:

```bash
python3 scripts/docs_inventory.py --converter markitdown
python3 scripts/docs_inventory.py --converter docling
python3 scripts/docs_inventory.py --converter auto
```

3. Retrieval indexing

Build a LlamaIndex-style index from:

- `docs/generated/documentation_intelligence/docs_inventory.json`
- Markdown source files
- converted Markdown from Docling or MarkItDown
- OpenAPI and JSON evidence files when relevant

Use hybrid retrieval where possible:

- lexical search for exact filenames, contracts, status labels, commands, and evidence identifiers
- vector search for concepts, summaries, architecture themes, and cross-document synthesis

Every generated higher-level document must cite source paths from the inventory.

4. Higher-level document generation

Use `docs/generated/documentation_intelligence/docs_generation_plan.md` as the queue of candidate deliverables. Generate each document from source paths listed in the plan and from retrieval over the indexed corpus.

Expected first-pass deliverables include:

- System Overview
- Architecture Handoff
- Production Readiness Report
- Roadmap Consolidation
- Audit Evidence Index
- Developer Onboarding Guide
- API Documentation Index
- POPIA Compliance Pack
- Beta Launch Pack
- CAPS Content Coverage Pack

5. Final artifact rendering

Choose the artifact skill by deliverable type:

- Use `documents:documents` for polished `.docx` reports, handoff packs, compliance packets, and formal review documents.
- Use `presentations:Presentations` for stakeholder, investor, launch, or executive decks.
- Use `spreadsheets:Spreadsheets` for evidence matrices, coverage matrices, task inventories, readiness checklists, and traceability tables.
- Use `hugging-face:huggingface-papers` only for arXiv or Hugging Face paper analysis, such as research references, model papers, linked datasets, linked Spaces, or paper metadata. It is not the primary crawler for EduBoost's internal docs.

## Automation Commands

Generate or refresh inventory artifacts:

```bash
python3 scripts/docs_inventory.py
```

Check whether generated artifacts are current:

```bash
python3 scripts/docs_inventory.py --check
```

If Makefile targets are available:

```bash
make docs-inventory
make docs-inventory-check
```

## Quality Rules

- Generated summaries must point back to concrete source files.
- Prefer local repository sources over memory or external search.
- Treat status terms such as `TODO`, `DRAFT`, `BLOCKED`, `FAIL`, and `needs review` as risk signals until reconciled.
- Treat evidence, contracts, manifests, attestations, and checklists as source material for audit-grade deliverables.
- Do not overwrite human-authored docs with generated text unless the request explicitly asks for that promotion.
- Keep generated artifacts under `docs/generated/documentation_intelligence/` unless a document is intentionally promoted into the canonical docs tree.

## Dependency Guidance

The base inventory script uses the Python standard library so it can run in CI without additional packages.

Optional packages for richer workflows:

```bash
pip install markitdown
pip install docling
pip install llama-index
```

Prefer adding these to a separate docs intelligence environment or optional requirements file unless the project decides to make them part of the default docs toolchain.
