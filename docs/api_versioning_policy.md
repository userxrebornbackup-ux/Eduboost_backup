# EduBoost API Versioning Policy

## Current Version

The current production API version is:

```text
v2
```

The canonical runtime is:

```text
app.api_v2:app
```

## Supported Prefixes

During the current migration window, production V2 routers are exposed under both prefixes:

```text
/api/v2
/v2
```

## Versioning Rules

- New production API work must target V2.
- New V1 routes are forbidden.
- Archived V1 compatibility behavior must not be mounted in `app.api_v2:app`.
- Breaking API changes require an explicit PR label and OpenAPI diff review.
- `docs/openapi.json` must be regenerated when route, request, or response contracts change.
- `make openapi-check` must pass before merge.

## Compatibility Policy

`app.legacy.api.main:app` may remain as a compatibility import shim, but it is not the canonical production runtime.

Known legacy compatibility behavior:

- `/api/v1/lessons/generate` may return HTTP 410 Gone when the legacy shim is explicitly imported.
- That route must not appear in the canonical V2 OpenAPI schema.

## Evidence

- Runtime tests: `tests/test_entrypoints.py`
- Legacy exclusion tests: `tests/test_legacy_route_exclusion.py`
- OpenAPI generator: `scripts/generate_openapi.py`
- Drift check: `make openapi-check`

## PR-002R Formal Versioning Policy Addendum

# API Versioning Policy

**Applies to:** EduBoost SA API V2 (`app.api_v2:app`)
**Version of this policy:** 1.0
**Relates to:** PR-002R §1.7

---

## Principles

1. **Stability first.** Clients (the Next.js frontend and any third-party
   integrations) must not be broken by a server-side change.
2. **Explicit versioning.** The API version is encoded in the URL path prefix
   (`/api/v2/...`) rather than in headers or query params.
3. **Envelope stability.** The `data` / `error` / `meta` envelope structure is
   a contractual guarantee for the lifetime of V2.

---

## Change Classification

| Change type | Backward compatible? | Action required |
|---|---|---|
| Add new optional field to a response | ✅ Yes | None — release normally |
| Add new endpoint | ✅ Yes | None — release normally |
| Add new optional query/body param with default | ✅ Yes | None — release normally |
| Add new error code | ✅ Yes | Document in `docs/error_contract.md` |
| Rename or remove a field | ❌ No | See **Breaking Changes** below |
| Change field type | ❌ No | See **Breaking Changes** below |
| Remove an endpoint | ❌ No | See **Breaking Changes** below |
| Change HTTP method of an endpoint | ❌ No | See **Breaking Changes** below |
| Change envelope structure | ❌ No | Requires major version bump |

---

## Non-Breaking Changes

Deploy freely.  Update `CHANGELOG.md` and `docs/openapi.json`.

CI will catch unintended schema drift via the `openapi-contract` job.

---

## Breaking Changes

1. **Open a GitHub issue** labelled `breaking-change` at least **90 days**
   before the change lands in production.
2. Add a deprecation notice to the affected endpoint's OpenAPI `description`
   field.
3. Implement the change under a **new version prefix** (e.g. `/api/v3/...`)
   while keeping the old endpoint alive for the deprecation period.
4. After the deprecation period, remove the old endpoint and update
   `docs/legacy_compatibility.md`.
5. Breaking-change PRs require the `breaking-api-change` label on GitHub
   before merge.

---

## Deprecation Process

1. Add `Deprecated: true` to the endpoint's OpenAPI metadata.
2. Add a `Sunset` response header with the planned removal date.
3. Log a warning server-side whenever the deprecated endpoint is called.
4. Announce in `CHANGELOG.md` under a `Deprecated` section.

---

## Version Lifecycle

| Version | Prefix | Status |
|---|---|---|
| V1 | `/api/v1/*` | **Archived** — not exposed by `app.api_v2:app` |
| V2 | `/api/v2/*` | **Current** — all new development |
| V3 | `/api/v3/*` | Not yet planned |

---

## Schema Governance

- `docs/openapi.json` is the source of truth for the V2 schema.
- Generated by `scripts/generate_openapi.py` from the live app.
- Checked in CI by the `openapi-contract` job.
- Schema hash is included in every release evidence bundle.
- Breaking-change detection is done by diffing `paths` and `components` keys
  between the previous and current committed schema.
